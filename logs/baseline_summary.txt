========================================================================
RERANKING SETUP
========================================================================
  What is reranking?
    Standard retrieval (first stage) ranks documents by vector similarity —
    the cosine distance between the query embedding and each document
    embedding. This is fast but coarse: a single vector must capture the
    full meaning of a query and a document.

    A cross-encoder reranker (second stage) takes the raw query text and
    each candidate document text and feeds them together through a
    transformer. Because it sees both at once, it can model fine-grained
    relevance (exact term overlap, entity matching, logical entailment)
    that embedding similarity misses.

  Pipeline used here (two-stage):
    1. Flat FAISS (IndexFlatIP, exact) retrieves top-50 candidates by
       embedding similarity — same index as the flat baseline.
    2. BAAI/bge-reranker-v2-m3 (cross-encoder, ~570M params, fp16, GPU)
       scores all 50 (query, document) pairs and re-orders them.
    3. Top-10 reranked results are returned to the agent.

  Model : BAAI/bge-reranker-v2-m3
    - Architecture : XLM-RoBERTa-large fine-tuned for passage reranking
    - Parameters   : ~570M (fp16 → ~1.1 GB GPU memory)
    - Max length   : 512 tokens per (query, doc) pair
    - Languages    : Multilingual (covers BrowseComp+ diverse sources)
    - Output       : Single relevance logit per pair; higher = more relevant

  Result: Flat ≈ IVFSQ8 >> Reranking. Flat beats reranking at every K by
  19–49%. IVFSQ8 is within rounding error of flat at all K values.

  Why reranking hurts on BrowseComp+:
  1. Cross-encoder max_length=512 tokens truncates long multi-constraint
     questions, discarding the exact constraints that identify the gold doc.
  2. bge-reranker-v2-m3 was trained on standard IR (short factoid queries,
     Wikipedia passages) — not the deliberate obfuscation of BrowseComp+.
  3. Qwen3-Embed-8B is already a strong 8B embedding model; the cross-encoder
     cannot improve on its dot-product ranking for this domain.

  Reconciling with earlier agent-eval finding:
  - Agent eval Q1–Q40 showed reranking gold_recall=0.639 > flat=0.577.
  - That was noise: n=40 is too small for a reliable comparison.
  - The n=830 offline eval is the definitive measurement.
  - Both flat and reranking achieve ~44% end-to-end accuracy, confirming
    that retrieval quality is not the bottleneck — reasoning is.

  Note on agent vs single-query recall:
  - Single query: flat Recall@10=0.258 vs agent flat gold_recall=0.567
  - The agent's ~10.5 avg iterations with reformulated queries improves
    recall by 2.2× over a single query. Query reformulation matters more
    than reranking for this benchmark.

  Conclusion: Drop reranking. Stick with FAISS IndexFlatIP. Focus
  improvements on reasoning quality (prompt engineering, better LLM).
